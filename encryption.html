Encryption Overview

Tunneling encompasses Data Encapsulation and Encryption, but the way VPNs encrypt OSI information largely depends on the VPN provider and what people consider secure.

VPNs require 3 types of encryption:

Symmetric Encryption for protecting the transmitted data
Public Key Encryption for symmentric key distrbution
Hashing for Data Integrity

Symmetric Encryption

Symmetric Encryption is an encryption algorithm where encryption and decryption use the same key. The Advanced Encryption Standard (AES) is the most popular amongst reputable VPN services.

Some configurations of AES include:



-128 bit Key with 10 rounds
-192 bit Key with 12 rounds
-256 bit Key with 14 rounds





VPNs that use AES include:
ExpressVPN, NordVPN, Surfshark, CyberGhost, IPVanish, PrivateVPN, VyprVPN, ZenMate, PureVPN, StrongVPN, VPNArea, SaferVPN, Ivacy, GooseVPN, Windscribe, and HideMyAss (Anon).

Public Key Encryption

Public Key Encryption is used to ensure both the sender and receiver have the same key required for Symmetric Encrypted communication between your device and a VPN server.

In order to connect to a VPN server, two phases must occur:

-Session Establishment
-Transfer of AES keys

Session Establishment authenticates both the sender and receiver, as well as share information that assists in transferring AES keys. The routines that encompass Session Establishment are handled by a system called Transport Layer Security (Anon).

Transport Layer Security (Trappe and Washington)

Transport Layer Security (TLS) is the successor to Secure Socket Layer(SSL) which is why they are commonly interchanged when discussed in detail.

TLS can be broken into a Record Protocol, and Collection of Management Protocol. The Collection of Management Protocol sets up and manages the information required for the Record Protocol to occur. A full breakdown of these Protocols is here.

A major part of TLS is ensuring the authenticity of the sender and receiver's public keys. A user can have many public keys that are used for communication across the Internet so an established authoritative body to organize and manage users public keys became necessary: these bodies are called Certification Authorities (CA).

CAs maintain an infrastructure that associates public keys with a user in an entity called a Digital Certificate. The most commonly use certificate is called X.509 Certificate. Both sender and receiver acquire each other's certificates to access each other's public keys instead of asking each other. These certificates include not only the public key, but also a digital signature and signature algorithm that can be used to verify the public key.

Diffie-Hellman Key Exchange (DF)

DH may be used by VPNs for session establishment as it relies on the Discrete Log Problem for secure public keys. DH requires both the sender and receiver to make a random contribution to the key establishment. Since the connection is occurring between a user and a server, the server's random contribution can be set, or made static, and put on their Digital Certificate, or can be newly generated for every sign-on, or made Ephemeral (Anon).

Hashing

SHA(Anon)

For Hashing Algorithms, the most common is the SHA-2 family, consisting of SHA-256, SHA-384, and SHA-512.

VPN Services that use SHA-256: CyberGhost, PrivateVPN, VyprVPN, ZenMate, PureVPN, VPNArea, SaferVPN, and HideMyAss.

VPN Services that use SHA-384: NordVPN

VPN Services that use SHA-512: ExpressVPN, IPVanish, Surfshark, StrongVPN, and Windscribe.

Other Notes on Security (Anon)

Varying Key Lengths

Some VPN services use the shorter key length versions of their encryption algorithms for different platforms to improve performance.
For instance, NordVPN's desktop application uses AES-256 while NordVPN's browser extension uses AES-128.

While AES may be the Symmetric Encryption of choice, some VPNs use more encryption algorithms on top of the AES output for more security. These extra measures include Cipher Block Chaining (CBC) and Galois/Counter Mode (GCM).

Cipher Block Chaining (CBC)

CBC encryption lies in breaking up the text into blocks and encrypting the XOR of the current block's text and the last block's cipher text. This slows down the VPN encryption process as the encryptions of each block cannot be done in parallel.

Galois/Counter Mode(GCM).

GCM is very similar to CBC in that encryption of a block lies in XORing. The main difference is that a random number is generated, encrypted, XORed with the current text, then incremented for the next block of text. Since the encryption of each block depends on the counter, parallel encryption of blocks can occur which prevents VPN encryption slow-down.

Encryption Scheme for CBC.
C0 is initalized by the sender. (Trappe and Washington 304)

Encryption Scheme for GCM.
The Counter is initialized to a random number and increments for every block encrypted. ("Galois/Counter Mode")
